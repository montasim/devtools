EditorPane Architecture - Which Approach?

All options use CodeMirror editors + Shiki highlighting. The difference is in component structure and data flow.

Approach A: Monolithic Component

Single EditorPane.tsx - ~400 lines

  • Both editors + diff panel in one file
  • Direct state management with useState
  • Simple, easy to understand
  • Harder to test individual pieces
  • Becomes unwieldy as features grow
File Structure
components/editor-pane/
├── editor-pane.tsx (everything)
├── types.ts
└── index.ts

Approach B: Modular Components ✓

Split concerns - 4-5 focused files

  • EditorPane (orchestrator)
  • DiffPanel (unified diff view)
  • JsonEditor (CodeMirror wrapper)
  • useJsonDiff (custom hook)
  • Each piece testable in isolation
  • Follows your navbar/toolbar pattern
File Structure
components/editor-pane/
├── editor-pane.tsx (orchestrator)
├── diff-panel.tsx
├── json-editor.tsx
├── use-json-diff.ts
├── types.ts
└── index.ts

Approach C: Feature-Based Modules

Separate by capability - 6-8 files

  • EditorPane + JsonEditor (UI)
  • diff-engine/ (logic)
  • json-parser/ (validation)
  • formatters/ (pretty print)
  • Most flexible for future formats
  • More boilerplate now
File Structure
components/editor-pane/
├── editor-pane.tsx
├── json-editor.tsx
├── diff-panel.tsx
├── lib/
│ ├── diff-engine/
│ ├── json-parser/
│ └── formatters/
└── types.ts

Recommendation: Approach B (Modular)

Why: Matches your existing navbar/toolbar architecture pattern. Splits concerns without over-engineering. Each component has a single responsibility. Easy to test now, easy to add XML/CSV formats later.
