# DevTools v2 — Clean Architecture Rewrite Plan

## Branch: `refactor/v2-clean-architecture`

---

## 1. Problem Statement

The current codebase has significant code duplication across the three tool pages (`/json`, `/text`, `/base64`):

- **3 nearly identical save-utils** (only differ in `pageName` and tool-to-tab mapping)
- **10+ share dialogs** with identical structure (Sheet + ShareForm + Copy/Download section)
- **Duplicated localStorage patterns** in every tab (shared data loading, debounced save, mounted state)
- **Duplicated toolbar/action patterns** (Clear/Save/Share buttons repeated per tab)
- **Duplicated copy/download utilities** scattered across components
- **Business logic mixed into UI components** (tab files are 150-600 lines doing too much)

## 2. Architecture Principles

### SOLID

- **Single Responsibility**: Each module handles one concern (storage, sharing, diff, formatting)
- **Open/Closed**: New tools added via configuration, not by duplicating components
- **Liskov Substitution**: Generic tool components work with any tool type via interfaces
- **Interface Segregation**: Small, focused interfaces per consumer
- **Dependency Inversion**: Components depend on abstractions (hooks/interfaces), not concrete implementations

### Clean Code

- No duplicate code — extract into shared utilities/hooks
- Small, focused functions (<30 lines)
- Meaningful naming
- Consistent patterns across the codebase

### Modular Approach

- Feature-based folder structure
- Clear separation: UI / Logic / Data / Types
- Plugin-style tool registration

#### 4.6.2 `useDownload()` — Replaces 45 download-as-file instances

```typescript
// lib/hooks/use-download.ts
export function useDownload() {
  const download = useCallback((content: string, filename: string, mimeType = 'text/plain') => {
    try {
      const blob = new Blob([content], { type: mimeType });
      const url = URL.createObjectURL(blob);
      const a = document.createElement('a');
      a.href = url;
      a.download = filename;
      a.click();
      URL.revokeObjectURL(url);
      toast.success(`Downloaded ${filename}`);
    } catch {
      toast.error('Failed to download file');
    }
  }, []);

  return { download };
}

// Usage:
```

**Replaces**: Every `new Blob` + `URL.createObjectURL` + download pattern across 29 files.

#### 4.6.3 `useLocalStorage()` — Replaces 72 localStorage get/set instances

```typescript
// lib/hooks/use-local-storage.ts
export function useLocalStorage<T>(key: string, initialValue: T) {
    const [storedValue, setStoredValue] = useState<T>(() => {
        if (typeof window === 'undefined') return initialValue;
        try {
            const item = window.localStorage.getItem(key);
            return item ? JSON.parse(item) : initialValue;
        } catch {
            return initialValue;
        }
    });

    const setValue = useCallback(
        (value: T | ((val: T) => T)) => {
            setStoredValue((prev) => {
                const valueToStore = value instanceof Function ? value(prev) : value;
                try {
                    window.localStorage.setItem(key, JSON.stringify(valueToStore));
                } catch (error) {
                    console.error(`Failed to save to localStorage key "${key}":`, error);
                }
                return valueToStore;
            });
        },
        [key],
    );

    const remove = useCallback(() => {
        try {
            window.localStorage.removeItem(key);
            setStoredValue(initialValue);
        } catch (error) {
            console.error(`Failed to remove localStorage key "${key}":`, error);
        }
    }, [key, initialValue]);

    return [storedValue, setValue, remove] as const;
}
```

**Replaces**: Every `localStorage.getItem` / `localStorage.setItem` + try/catch pattern across 19 files.

## 6. API Route Redesign

### Current Issues

- Inconsistent error response shapes
- 3 separate save-utils calling the same endpoint with slightly different bodies
- No proper middleware for auth
- Rate limiting is in-memory

### Redesigned API Structure

```
POST /api/v1/auth/login
POST /api/v1/auth/logout
GET  /api/v1/auth/me
POST /api/v1/auth/register/send-otp
POST /api/v1/auth/register/verify-otp
POST /api/v1/auth/password-reset/send-otp
POST /api/v1/auth/password-reset/confirm
POST /api/v1/auth/update-password
POST /api/v1/auth/update-name

POST   /api/v1/saved              # Create saved item
GET    /api/v1/saved?pageName=    # List user's saved items
DELETE /api/v1/saved/:id          # Delete saved item

POST   /api/v1/shares             # Create share
GET    /api/v1/shares?pageName=   # List user's shares
GET    /api/v1/shares/:id         # Get share metadata
POST   /api/v1/shares/:id/access  # Access shared content (with password)
DELETE /api/v1/shares/:id         # Delete share
```

### Standardized Response Shapes

```typescript
// Success response
{ ok: true, data: T }

// Error response
{ ok: false, error: { code: string, message: string, details?: unknown } }
```

## 11. Full Duplication Audit (Quantified)

This section documents every duplicated pattern discovered during analysis, forming the basis for all generic reusable utilities.

### 11.1 Copy to Clipboard — 62 instances across 39 files

Every instance follows this exact pattern:

```typescript
try {
    await navigator.clipboard.writeText(text);
    toast.success('Copied to clipboard');
} catch (error) {
    console.error('Failed to copy:', error);
    toast.error('Failed to copy to clipboard');
}
```

**Replacement**: `useClipboard()` hook — one line per usage: `copy(text, successMessage?)`

### 11.2 Download as File — 45 instances across 29 files

Every instance follows this exact pattern:

```typescript
try {
    const blob = new Blob([content], { type: mimeType });
    const url = URL.createObjectURL(blob);
    const a = document.createElement('a');
    a.href = url;
    a.download = filename;
    a.click();
    URL.revokeObjectURL(url);
    toast.success('Downloaded...');
} catch (error) {
    console.error('Failed to download:', error);
    toast.error('Failed to download file');
}
```

**Replacement**: `useDownload()` hook — one line per usage: `download(content, filename, mimeType?)`

### 11.3 Share Dialog — 10+ near-identical components (~2400 total lines)

All 10 share dialogs share this identical structure:

```
Sheet (sidebar)
  ├── SheetHeader (title)
  ├── ShareForm (title, comment, expiration, password inputs)
  ├── ShareResult (generated URL + copy button)
  ├── Separator
  └── Tool-specific actions (copy content, download, export)
```

**Replacement**: `ShareSidebarModal` plugin with `extraActions` slot for tool-specific actions.

### 11.4 Save Utilities — 3 identical files (~230 total lines)

`lib/json-save-utils.ts`, `lib/text-save-utils.ts`, `lib/base64-save-utils.ts` differ only in:

- `pageName` string
- `toolToTabMap` mapping object
- `contentState` key (`leftContent` vs `rightContent`)

**Replacement**: Single `saveToolContent(pageName, toolName, content)` function.

### 11.5 Clear Dialog Pattern — 17 files with identical boilerplate

Every tab component contains:

```typescript
const [showClearDialog, setShowClearDialog] = useState(false);
const handleClearClick = () => setShowClearDialog(true);
const handleConfirmClear = () => {
    setShowClearDialog(false);
    onClear();
};
// + <ConfirmDialog open={showClearDialog} onOpenChange={setShowClearDialog} .../>
```

**Replacement**: `useClearDialog(onConfirm)` hook returns `{ open, setOpen, openDialog, confirm }`.

### 11.6 Shared Data Loading — 11 files with identical async pattern

Every tab that receives shared data has:

```typescript
const sharedDataLoadedRef = useRef(false);
const sharedDataRef = useRef(sharedData);

useEffect(() => {
    if (
        sharedData?.tabName === 'xxx' &&
        sharedData?.state?.leftContent &&
        !sharedDataLoadedRef.current
    ) {
        sharedDataLoadedRef.current = true;
        setTimeout(() => setLeftContent(sharedData.state?.leftContent || ''), 0);
    }
    sharedDataRef.current = sharedData;
}, [sharedData]);
```

**Replacement**: `useToolState()` hook handles sharedData, localStorage init, and mounted lifecycle.

### 11.7 Content Tracking useEffect — 12 files with redundant state

```typescript
const [currentLeftContent, setCurrentLeftContent] = useState('');
const [currentRightContent, setCurrentRightContent] = useState('');

useEffect(() => {
    setCurrentRightContent(rightContent);
}, [rightContent]);
useEffect(() => {
    setCurrentLeftContent(leftContent);
}, [leftContent]);
```

This is completely redundant — the values are already available as `leftContent`/`rightContent`.

**Replacement**: Remove the redundant state entirely. Pass values directly to share dialog.

### 11.8 Toast Messages — 199 instances with inconsistent wording

| Message                         | Variations           | Files |
| ------------------------------- | -------------------- | ----- |
| `'Copied to clipboard'`         | 6 different wordings | 20+   |
| `'Failed to copy to clipboard'` | 4 different wordings | 15+   |
| `'No content to share...'`      | 5 different wordings | 8+    |
| `'Failed to download...'`       | 3 different wordings | 10+   |
| `'Saved successfully!'`         | 1                    | 3     |
| `'Failed to save content'`      | 1                    | 3     |

**Replacement**: Standardized messages inside `useClipboard()`, `useDownload()`, `saveToolContent()`. Consistent wording across the entire app.
